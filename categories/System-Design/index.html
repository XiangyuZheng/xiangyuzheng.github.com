System Design


About System Design

It’s an open-ended conversation.

Topics:

  1. Concurrency. Do you understand threads, deadlock, and starvation? Do you know how to parallelize algorithms? Do you understand consistency and coherence?
  2. Networking. Do you roughly understand IPC and TCP/IP? Do you know the difference between throughput and latency, and when each is the relevant factor?
  3. Abstraction. You should understand the systems you’re building upon. Do you know roughly how an OS, file system, and database work? Do you know about the various levels of caching in a modern OS?
  4. Real-World Performance. You should be familiar with the speed of everything your computer can do, including the relative performance of RAM, disk, SSD and your network.
  5. Estimation. Estimation, especially in the form of a back-of-the-envelope calculation, is important because it helps you narrow down the list of possible solutions to only the ones that are feasible. Then you have only a few prototypes or micro-benchmarks to write.
  6. Availability and Reliability. Are you thinking about how things can fail, especially in a distributed environment? Do know how to design a system to cope with network failures? Do you understand durability?

You are not expected to give a perfect answer, there is no right answers. The process of how you solve the problem is more important.

Design the Facebook news feed function

What are the best practices for building something like a News Feed?
What are the scaling issues to keep in mind while developing a social network feed?

Users in most social networking sites are describable in terms of a social graph. The relationships between users are represented by adjacency lists. A connection is an edge in the graph.

Usually, we want to rank edges for one person.

#Database table

  1. User
    id
    name
    age
    …
    all_followers
    news_feed_content

  2. Activity
    id
    user_id
    source_id
    type (photo, comment, post)

  1. Follow
    id
    user_id
    follower_id

#Push model

When someone published a post, push the activity id to all its followers.
When read news feed, a user just need to get all content from news_feed_content.

drawbacks: For celebrities, there are too many to push, it needs many disk seeks, and write back-logs (write post ids).

#Pull model

Stores all recent activities in memory, in this way, we avoid all disk seeks.
Do nothing when publish a post.
When read news feed, we pull all the activities from recent activities in memory (Redis or Memcached)

drawbacks: when fails, it generate nothing to news feed. We may need a mechanism to approximates the user’s feed, e.g. just generate a subset of your friends.

#Hybrid

For celebrities, use Pull.
For normal users, use Push.

Shorten Url

System Design for Big Data-tinyurl

#On single machine:

DB table:
id
long url
short url

For every long url, we firstly insert it into the table, get its id. We then map the id into a 62-base numeric. [0-9a-zA-Z].

#On multiple machines:

We can use a distributed database to store the urls, and the DB can generate id for us.
Or we can do it ourselves.

The basic process can be:
Insert

  1. Hash an input long url into a single integer;
  2. Locate a server on the ring (consistent hashing) and store the key–longUrl on the server; The servers generate different ranges of ids.
  3. Compute the shorten url using base conversion (from 10-base to 62-base) and return it to the user.

Retrieve

  1. Convert the shorten url back to the key using base conversion (from 62-base to 10-base);
  2. Locate the server containing that key and return the longUrl.
